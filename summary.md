# Scrubbed result summary

Date of commitment: **2026-05-27 UTC**

## Benchmark

- SWE-bench Pro (Scale AI), Public set (`sweap_eval_full_v2.jsonl`).
- Leaderboard: https://labs.scale.com/leaderboard/swe_bench_pro_public

## Coverage

- Instances covered in the committed bundle: **53**
- Repository covered: **`internetarchive/openlibrary`** (single repo)
- Setting: closed-book at problem-solving time (worker did not have
  access to the curator's hidden test files when generating each patch).

## Cleanroom verdict distribution

| Verdict | Count |
|---|---:|
| PASS | 52 |
| TOOLING_ERROR | 1 |

Verdicts come from a deterministic cleanroom pipeline: pull the
benchmark's published Docker image, apply the patch, run `pytest` for
both the `fail_to_pass` and `pass_to_pass` test groups, record exit
codes. No LLM is involved in producing a verdict.

The one TOOLING_ERROR is a known infrastructure issue (dataset f2p test
ids were truncated past the pathspec validation step), not a
substantive disagreement on patch correctness. We disclose it rather
than dropping it.

## What this is not

This is **not a leaderboard submission**, and the 52/53 ratio is **not
directly comparable** to public leaderboard resolve rates:

- It covers a single repository, not the full 731-instance Public set.
- It uses a multi-attempt agent protocol, not the single-shot
  evaluations used to rank models on the public leaderboard.
- Subsetting and protocol differences are both selection effects that
  pull the visible rate higher than a single-shot full-benchmark run
  would produce on the same model lineage.

These caveats will appear verbatim in the full bundle's README when
released.

## What is being released later

When the held archive is published, each of the 53 instances will be
accompanied by:

- `patch.diff`: the AI-generated source-only patch.
- `verdict.json`: the cleanroom test verdict, including exit codes,
  duration, and pytest output tails for both the `fail_to_pass` and
  `pass_to_pass` test groups.
- `meta.json`: the public benchmark identifiers (`instance_id`,
  `base_commit`, `container_image`, language, repo, test counts) that
  let any independent reviewer reproduce the verdict using only public
  data and a Docker runtime.

The exact bytes that will be released are pinned by
[`hash.txt`](hash.txt).
